# 12 - 导航栏设计指南

> 导航是用户与产品之间的桥梁。好的导航让用户毫不费力地找到他们需要的内容。

---

## 为什么导航设计如此重要？

导航栏是用户访问网站或应用时最先接触的交互元素之一。研究表明：

- **94%** 的用户认为"易于导航"是网站最重要的特征
- 用户平均在 **3 次点击内** 找不到目标就会离开
- 糟糕的导航是用户放弃网站的 **第一大原因**

> ✨ 导航设计的黄金法则：**不要让用户思考。**

---

## 导航栏的类型

### 1. 顶部导航栏 (Top Navigation Bar)

**特点**：
- 水平排列在页面顶部
- 最常见的导航模式
- 通常包含 Logo、主要链接、搜索和用户操作
- 适合 5-7 个主要导航项

**结构示例**：
```
┌─────────────────────────────────────────────────────────┐
│ [Logo]    首页  产品  方案  资源  定价    [搜索] [登录] │
└─────────────────────────────────────────────────────────┘
```

**适用场景**：
- 企业官网
- SaaS 产品
- 电商平台
- 新闻/内容网站

**优点**：
- 用户最熟悉的模式，学习成本低
- 主要页面一目了然
- 符合从左到右的阅读习惯

**缺点**：
- 导航项数量受水平空间限制
- 移动端需要转换为其他形式

**参考网站**：
- [Stripe](https://stripe.com) — 简洁优雅的顶部导航，悬停展开子菜单
- [GitHub](https://github.com) — 功能丰富但层次清晰的顶部导航
- [Airbnb](https://airbnb.com) — 搜索为中心的顶部导航设计

---

### 2. 侧边导航栏 (Sidebar Navigation)

**特点**：
- 垂直排列在页面左侧（偶尔右侧）
- 支持大量导航项和多级嵌套
- 通常可折叠/展开
- 适合复杂应用和管理后台

**结构示例**：
```
┌──────────┬──────────────────────────┐
│ [Logo]   │                          │
│          │                          │
│ 📊 仪表盘│       主内容区域          │
│ 👥 用户  │                          │
│  ├ 列表  │                          │
│  └ 分组  │                          │
│ 📦 项目  │                          │
│ ⚙️ 设置  │                          │
│          │                          │
└──────────┴──────────────────────────┘
```

**适用场景**：
- 管理后台 (Dashboard)
- 数据分析平台
- 文档网站
- 企业级应用 (B2B SaaS)

**优点**：
- 可容纳大量导航项
- 支持多级层次结构
- 内容区域垂直空间不受影响
- 可折叠以节省空间

**缺点**：
- 占用水平空间
- 移动端适配需要特殊处理
- 层级过深时用户容易迷失

**参考网站**：
- [Notion](https://notion.so) — 可折叠的树形侧边导航
- [Linear](https://linear.app) — 图标 + 文字的紧凑侧边栏
- [Figma](https://figma.com) — 上下文感知的侧边导航
- [Discord](https://discord.com) — 双层侧边栏（服务器 + 频道）

---

### 3. 底部导航栏 (Bottom Navigation Bar)

**特点**：
- 固定在屏幕底部
- 主要用于移动端应用
- 通常 3-5 个图标 + 文字标签
- 拇指可轻松触达（符合拇指热区）

**结构示例**：
```
┌─────────────────────────────────────┐
│                                     │
│            主内容区域                │
│                                     │
├─────────────────────────────────────┤
│  🏠首页   🔍发现   ➕   💬消息   👤我 │
└─────────────────────────────────────┘
```

**适用场景**：
- 移动端 App（iOS / Android）
- PWA（渐进式 Web 应用）
- 移动优先的网站

**优点**：
- 拇指友好，操作方便
- 主要功能始终可见
- 符合移动端用户习惯

**缺点**：
- 空间有限，最多 5 个项
- 桌面端使用不自然
- 占用宝贵的屏幕底部空间

**设计规范**：

| 平台 | 图标尺寸 | 栏高度 | 最少项数 | 最多项数 |
|------|---------|--------|---------|---------|
| iOS (Tab Bar) | 25×25pt | 49pt | 3 | 5 |
| Android (Navigation Bar) | 24×24dp | 56dp-80dp | 3 | 5 |
| Web (Bottom Nav) | 24×24px | 56px | 3 | 5 |

**参考网站/应用**：
- [Instagram](https://instagram.com) — 经典五图标底部栏
- [YouTube](https://youtube.com) — 中间突出的创建按钮
- [Spotify](https://spotify.com) — 简洁三栏底部导航

---

### 4. 汉堡菜单 (Hamburger Menu ☰)

**特点**：
- 用三条横线图标 (☰) 隐藏导航内容
- 点击/触摸后展开完整菜单
- 节省屏幕空间
- 适合次要导航或移动端

**展开方式**：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| 侧滑抽屉 (Drawer) | 从左侧或右侧滑出 | 移动端最常见 |
| 全屏覆盖 (Full-screen Overlay) | 占据整个屏幕 | 品牌/创意网站 |
| 下拉面板 (Dropdown Panel) | 从顶部向下展开 | 桌面端使用 |

**适用场景**：
- 移动端的辅助导航
- 导航项较多的响应式网站
- 创意/作品集网站
- 需要最大化内容区域的页面

**优点**：
- 极大节省屏幕空间
- 可容纳大量导航项
- 界面看起来更简洁

**缺点**：
- 隐藏导航降低了可发现性（discoverability）
- 增加一次额外点击
- 部分用户（尤其是非技术用户）不理解 ☰ 图标

**参考网站**：
- [Awwwards](https://awwwards.com) — 全屏覆盖式汉堡菜单
- [Medium](https://medium.com) — 移动端简洁的侧滑抽屉

---

### 5. 面包屑导航 (Breadcrumb Navigation)

**特点**：
- 显示用户当前位置的路径层级
- 辅助导航，不作为主导航使用
- 帮助用户理解网站结构

**结构示例**：
```
首页 > 产品 > 手机 > iPhone 16 Pro
```

**适用场景**：
- 电商网站（商品分类层级）
- 文档网站（多层目录）
- 企业级应用（复杂层级）

**参考网站**：
- [Amazon](https://amazon.com) — 商品分类面包屑
- [Apple](https://apple.com) — 简洁的产品路径导航

---

### 6. 标签导航 (Tab Navigation)

**特点**：
- 在同一页面内切换不同视图/内容
- 水平或垂直排列
- 适合平级关系的内容分组

**结构示例**：
```
┌──────┬──────┬──────┐
│ 概览 │ 分析 │ 设置 │ ← 激活状态有下划线/高亮
├──────┴──────┴──────┤
│                    │
│    当前标签内容     │
│                    │
└────────────────────┘
```

**适用场景**：
- 页面内分区（如个人资料页的"动态/关注/粉丝"）
- 数据面板切换
- 设置页面分组

**参考网站**：
- [GitHub](https://github.com) — 仓库页的 Code/Issues/PR 标签切换
- [Twitter/X](https://x.com) — 个人主页的推文/回复/喜欢标签

## 导航设计最佳实践

### 1. 信息架构优先

在设计导航之前，先确定信息架构（IA）：

```
步骤 1：列出所有页面/功能
    ↓
步骤 2：卡片分类（Card Sorting）— 按用户心智模型分组
    ↓
步骤 3：确定层级关系（主导航 vs 子导航 vs 工具导航）
    ↓
步骤 4：控制深度（不超过 3 层）
    ↓
步骤 5：验证（用户测试 / 树测试）
```

### 2. 导航层级控制

| 层级 | 说明 | 展示方式 |
|------|------|---------|
| 第一层 | 核心功能入口（3-7 项） | 顶部导航 / 底部导航 |
| 第二层 | 功能分类（每组 5-9 项） | 下拉菜单 / 侧边栏子项 |
| 第三层 | 具体页面 | 面包屑 / 页面内导航 |

> ⚠️ **三次点击原则**：用户应在 3 次点击内到达任何页面。层级过深会导致用户流失。

### 3. 导航标签命名

**好的命名**：
- ✅ 使用用户语言，而非内部术语
- ✅ 简短明确（2-4 个字为佳）
- ✅ 名词优先（"产品""定价""文档"）
- ✅ 相互排斥，无歧义

**坏的命名**：
- ❌ "解决方案" — 太模糊
- ❌ "资源中心" vs "知识库" — 区别不清
- ❌ "更多" — 用户不知道里面有什么
- ❌ "杂项" — 说明分类有问题

### 4. 视觉状态设计

每个导航项应有明确的视觉状态：

| 状态 | 说明 | 设计要点 |
|------|------|---------|
| 默认 (Default) | 正常状态 | 清晰可读 |
| 悬停 (Hover) | 鼠标悬停 | 颜色变化或下划线 |
| 激活 (Active) | 当前页面 | 强调色、粗体或下划线 |
| 禁用 (Disabled) | 不可用 | 灰色、降低透明度 |
| 聚焦 (Focus) | 键盘聚焦 | 可见的焦点轮廓 |

**CSS 示例**：
```css
/* 顶部导航状态 */
.nav-link {
  color: #4a5568;
  padding: 8px 16px;
  transition: color 0.2s, border-bottom 0.2s;
  border-bottom: 2px solid transparent;
}

.nav-link:hover {
  color: #2d3748;
  border-bottom-color: #e2e8f0;
}

.nav-link.active {
  color: #3182ce;
  font-weight: 600;
  border-bottom-color: #3182ce;
}

.nav-link:focus-visible {
  outline: 2px solid #3182ce;
  outline-offset: 2px;
}
```

### 5. 固定导航 vs 滚动隐藏

| 策略 | 适用场景 | 实现方式 |
|------|---------|---------|
| 固定顶部 (Sticky) | 频繁需要导航操作 | `position: sticky; top: 0;` |
| 滚动隐藏 / 上滑显示 | 内容阅读为主，最大化阅读空间 | JavaScript 监听滚动方向 |
| 缩小固定 | 兼顾品牌展示和空间效率 | 滚动后 Logo 缩小、高度减小 |

## 响应式导航模式

### 断点策略

```
桌面端 (≥1024px)        平板 (768px-1023px)       移动端 (<768px)
┌────────────────┐     ┌────────────────┐      ┌────────────────┐
│ 完整顶部导航   │     │ 精简导航       │      │ 汉堡菜单       │
│ 所有项可见     │     │ 次要项折叠     │      │ + 底部导航栏   │
│ 下拉子菜单     │     │ 汉堡菜单可选   │      │ 核心3-5项固定  │
└────────────────┘     └────────────────┘      └────────────────┘
```

### 常见响应式模式

#### 模式 1：顶部导航 → 汉堡菜单

最常见的模式，桌面端展示完整导航，移动端折叠为汉堡菜单。

```css
/* 桌面端显示完整导航 */
.nav-menu {
  display: flex;
  gap: 24px;
}

/* 移动端切换为汉堡菜单 */
@media (max-width: 768px) {
  .nav-menu {
    display: none;
  }
  .nav-menu.open {
    display: flex;
    flex-direction: column;
    position: fixed;
    inset: 0;
    background: white;
    padding: 80px 24px 24px;
  }
  .hamburger-btn {
    display: block;
  }
}
```

#### 模式 2：侧边栏 → 底部抽屉

后台应用常用模式，桌面端侧边栏在移动端变为底部滑出抽屉。

#### 模式 3：优先级+导航 (Priority+ Pattern)

根据屏幕宽度动态显示导航项，放不下的自动归入"更多"菜单。

```
宽屏：  首页  产品  方案  资源  定价  博客  关于
中屏：  首页  产品  方案  资源  [更多 ▾]
窄屏：  首页  产品  [更多 ▾]
```

**优点**：比汉堡菜单更好的可发现性，用户始终能看到重要导航项。

#### 模式 4：混合导航

移动端同时使用顶部简化导航 + 底部固定导航栏。

```
┌────────────────────┐
│ [☰]  Logo  [🔍][👤]│ ← 顶部：品牌 + 工具操作
├────────────────────┤
│                    │
│     内容区域       │
│                    │
├────────────────────┤
│ 🏠  📂  ➕  💬  ⚙️ │ ← 底部：核心功能入口
└────────────────────┘
```

## 无障碍设计 (Accessibility)

### WCAG 导航要求

导航的无障碍设计不只是"做好事"，它直接影响用户覆盖面和法律合规。

### 1. 语义化 HTML

```html
<!-- ✅ 正确：使用 nav 元素和 ARIA 标签 -->
<nav aria-label="主导航">
  <ul role="menubar">
    <li role="none">
      <a href="/home" role="menuitem">首页</a>
    </li>
    <li role="none">
      <a href="/products" role="menuitem" aria-current="page">产品</a>
    </li>
    <li role="none">
      <a href="/pricing" role="menuitem">定价</a>
    </li>
  </ul>
</nav>

<!-- ❌ 错误：使用 div 代替语义元素 -->
<div class="nav">
  <div class="nav-item" onclick="goto('/home')">首页</div>
</div>
```

### 2. 键盘导航

| 按键 | 行为 |
|------|------|
| `Tab` | 在导航项之间移动焦点 |
| `Enter` / `Space` | 激活当前项 |
| `Escape` | 关闭子菜单 / 抽屉 |
| `→` / `←` | 水平导航中在项之间移动 |
| `↓` / `↑` | 打开/关闭子菜单，在子菜单项间移动 |
| `Home` / `End` | 跳到第一个/最后一个导航项 |

### 3. 跳过导航链接 (Skip Navigation)

为屏幕阅读器和键盘用户提供跳过重复导航的能力：

```html
<!-- 页面最顶部 -->
<a href="#main-content" class="skip-link">
  跳到主要内容
</a>

<nav aria-label="主导航">
  <!-- 导航内容 -->
</nav>

<main id="main-content">
  <!-- 页面主要内容 -->
</main>
```

```css
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  padding: 8px 16px;
  background: #3182ce;
  color: white;
  z-index: 100;
  transition: top 0.2s;
}

.skip-link:focus {
  top: 0;
}
```

### 4. ARIA 属性速查

| 属性 | 用途 | 示例 |
|------|------|------|
| `aria-label` | 为导航区域命名 | `<nav aria-label="主导航">` |
| `aria-current="page"` | 标识当前页面 | `<a aria-current="page">` |
| `aria-expanded` | 标识子菜单展开状态 | `<button aria-expanded="false">` |
| `aria-haspopup` | 标识有弹出子菜单 | `<button aria-haspopup="true">` |
| `role="navigation"` | 语义角色（`<nav>` 自带） | `<div role="navigation">` |

### 5. 对比度和触摸目标

- **文字对比度**：导航文字与背景的对比度至少 **4.5:1**（WCAG AA）
- **触摸目标**：移动端导航项最小点击区域 **44×44px**（WCAG 2.5.8）
- **焦点指示器**：焦点轮廓不能被 `outline: none` 移除，除非提供替代方案

## 常见导航错误与解决方案

### ❌ 错误 1：导航项过多

**问题**：把所有页面都放进主导航，导致选择困难。

**解决**：
- 主导航控制在 5-7 项
- 使用下拉菜单或子导航处理次级内容
- 定期审查分析数据，移除低频项

### ❌ 错误 2：在桌面端使用汉堡菜单

**问题**：桌面端有充足空间却仍隐藏导航，降低可发现性。

**解决**：
- 桌面端始终展示主导航
- 仅在移动端/平板端使用汉堡菜单
- 如必须使用，搭配关键 CTA 按钮外露

### ❌ 错误 3：没有当前位置指示

**问题**：用户不知道自己在哪个页面。

**解决**：
- 激活项使用不同颜色、粗体或下划线
- 添加 `aria-current="page"` 辅助屏幕阅读器
- 对于深层级页面，使用面包屑导航

### ❌ 错误 4：导航标签含糊不清

**问题**：使用"解决方案""平台""资源"等模糊词汇。

**解决**：
- 使用具体、可预测的标签
- 进行用户测试（树测试 / 首次点击测试）
- 确保标签之间无重叠

### ❌ 错误 5：移动端导航不考虑拇指热区

**问题**：重要操作放在屏幕顶部，拇指难以触及。

**解决**：
- 核心操作放在底部导航栏
- 使用拇指热区图指导布局
- 触摸目标不小于 44×44px

```
拇指热区分布（右手持握）：
┌──────────────────┐
│ 😫 困难区域      │
│                  │
│ 😐 可触及区域    │
│                  │
│ 😊 舒适区域      │
│ ████████████████ │ ← 底部导航最佳位置
└──────────────────┘
```

### ❌ 错误 6：忽略搜索作为导航方式

**问题**：对于内容丰富的网站，仅依赖分层导航。

**解决**：
- 在导航栏中加入搜索功能
- 支持搜索建议和自动补全
- 对于 100+ 页面的网站，搜索应与导航同等重要

## 导航类型选择指南

### 按项目类型选择

| 项目类型 | 推荐导航 | 原因 |
|---------|---------|------|
| 企业官网 | 顶部导航 + 大型下拉 | 内容分类清晰，品牌展示优先 |
| SaaS 产品 | 顶部导航 + 侧边栏 | 频繁操作需要快速切入功能 |
| 管理后台 | 侧边栏导航 | 功能多、层级深，侧边栏可折叠 |
| 移动 App | 底部导航 + 汉堡菜单 | 拇指友好，核心功能始终可见 |
| 电商网站 | 顶部导航 + 大型菜单 + 面包屑 | 商品分类多，用户需要定位当前位置 |
| 博客/内容 | 顶部简洁导航 | 内容为王，导航不喧宾夺主 |
| 作品集 | 极简导航或全屏菜单 | 突出视觉作品，导航尽量隐形 |
| 文档网站 | 侧边栏 + 面包屑 + 搜索 | 层级深，需要快速定位 |

### 决策树

```
你的产品是什么？

│   ├── 核心功能 ≤ 5 个 → 底部导航栏

└── 电商网站
    └── 顶部导航 + 大型菜单 + 面包屑
```

## 处理大型导航菜单（导航项非常多时的设计策略）

当网站或应用拥有数十甚至数百个导航项时，传统的 5-7 项顶部导航远远不够。本节介绍 7 种经过验证的策略，帮助你优雅地处理大规模导航。

> 💡 核心原则：**导航项越多，越需要结构化和渐进展示，而不是一次性全部呈现。**

---

### 1. 超级菜单 (Mega Menu)

**什么是超级菜单？**

超级菜单是一种大面积的下拉面板，通常占据页面宽度的大部分或全部，以多列布局展示大量分类和链接。

**结构示例**：
```
┌─────────────────────────────────────────────────────────────────┐
│ [Logo]    产品 ▾    解决方案    资源    定价         [搜索] [登录]│
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  📱 移动产品        💻 桌面产品        ☁️ 云服务        🎯 推荐  │
│  ─────────         ─────────         ─────────       ───────   │
│  iPhone            MacBook Air       iCloud          新品速递   │
│  iPad              MacBook Pro       Apple Music     限时优惠   │
│  Apple Watch       iMac              Apple TV+       教育优惠   │
│  AirPods           Mac Pro           Arcade                    │
│  配件              显示器             Fitness+                  │
│                                                                 │
│  [查看所有产品 →]                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**适用场景**：
- 电商网站（大量商品分类）
- 企业官网（多业务线产品矩阵）
- 新闻/媒体网站（多频道内容）
- SaaS 平台（功能模块丰富）

**设计要点**：

| 要素 | 最佳实践 |
|------|---------|
| 列数 | 3-5 列为佳，超过 5 列用户难以扫视 |
| 分组标题 | 每列有清晰的分类标题，加粗或用色彩区分 |
| 视觉辅助 | 可加图标、缩略图帮助用户快速识别 |
| 关闭机制 | 鼠标移出区域自动关闭，或提供明确关闭按钮 |
| 触发方式 | 桌面端悬停触发（需加 200-300ms 延迟防误触），移动端点击触发 |
| 底部 CTA | 提供"查看全部"链接兜底 |

**常见错误**：
- ❌ 面板内容过多导致需要滚动 — 超级菜单应一屏可见
- ❌ 没有鼠标移出延迟，轻微移动就关闭
- ❌ 移动端直接使用桌面端的超级菜单布局
- ❌ 分类之间没有视觉分隔，信息混成一团

**参考网站**：
- [Amazon](https://amazon.com) — 侧滑式超级菜单，分类极其丰富
- [Apple](https://apple.com) — 优雅的全宽超级菜单，带产品图片
- [Microsoft](https://microsoft.com) — 多层分类的超级菜单
- [Walmart](https://walmart.com) — 电商典型的分类超级菜单

### 2. 级联菜单 / 飞出菜单 (Cascading / Flyout Menus)

**什么是级联菜单？**

级联菜单是多级嵌套的子菜单，当用户悬停或点击父级菜单项时，子菜单从侧边"飞出"展示下一层内容。

**结构示例**：
```
┌─────────────┐
│ 电子产品  ▸ │──→ ┌──────────────┐
│ 服装鞋帽    │    │ 手机       ▸ │──→ ┌───────────────┐
│ 家居生活    │    │ 电脑平板     │    │ iPhone 16 Pro │
│ 食品生鲜    │    │ 智能穿戴     │    │ iPhone 16     │
│ 图书影音    │    │ 摄影器材     │    │ iPhone SE     │
└─────────────┘    │ 游戏设备     │    │ 配件          │
                   └──────────────┘    └───────────────┘
```

**适用场景**：
- 侧边栏导航中的多级分类
- 电商商品分类树
- 文件管理器 / 操作系统菜单
- 功能密集的专业工具

**设计要点**：

| 要素 | 最佳实践 |
|------|---------|
| 层级深度 | 最多 3 层，超过 3 层建议改用其他方案 |
| 子菜单方向 | 默认向右展开，空间不足时向左 |
| 安全三角 | 使用"安全三角区"防止鼠标斜移时子菜单意外关闭 |
| 展开指示 | 用箭头 (▸) 明确标识有子菜单的项 |
| 移动端适配 | 改为点击展开/折叠，不使用悬停飞出 |

**安全三角原理**：
```
            ┌──────────┐
 鼠标位置 ●╲│ 子菜单   │
            ╲          │
  安全三角 → ╲         │
              ╲        │
               ╲       │
                └──────┘

在这个三角区域内移动鼠标时，子菜单不会关闭。
Amazon 是该技术的经典实践者。
```

**常见错误**：
- ❌ 没有实现安全三角，鼠标斜移时子菜单闪烁消失
- ❌ 层级过深（4 层以上），用户在菜单间迷失
- ❌ 子菜单遮挡了父菜单的其他项
- ❌ 触摸设备上依赖悬停触发

**参考网站**：
- [Amazon](https://amazon.com) — 安全三角的经典实现
- [阿里巴巴 (1688.com)](https://www.1688.com) — 大型电商级联菜单
- [京东 (JD.com)](https://jd.com) — 侧滑 + 级联组合菜单
- [Windows 右键菜单](https://docs.microsoft.com) — 操作系统级联菜单的典型设计

### 3. 搜索式导航 (Search-based Navigation)

**什么是搜索式导航？**

当导航项数量极大（100+）或内容高度动态时，搜索成为比层级菜单更高效的导航方式。用户通过输入关键词直接定位目标，而非层层浏览。

**结构示例**：
```
┌─────────────────────────────────────────────────┐
│                  [🔍 搜索...]                    │
│                                                  │
│  输入: "设置"                                    │
│  ┌──────────────────────────────────────────┐   │
│  │ 🔧 账户设置        设置 > 账户           │   │
│  │ 🔔 通知设置        设置 > 通知           │   │
│  │ 🔒 隐私设置        设置 > 隐私与安全     │   │
│  │ 🎨 外观设置        设置 > 显示           │   │
│  │ ⌨️ 快捷键设置      设置 > 键盘快捷键     │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  最近访问: 个人资料 · 团队管理 · 计费            │
│  热门功能: 导出数据 · API 密钥 · 集成            │
└─────────────────────────────────────────────────┘
```

**常见形式**：

| 形式 | 说明 | 触发方式 | 代表产品 |
|------|------|---------|---------|
| Command Palette | 全局搜索 + 命令执行 | `Ctrl/Cmd + K` | VS Code, Figma, Linear |
| Spotlight Search | 全站内容搜索 | 搜索框或快捷键 | macOS, Notion, Confluence |
| Omnibox | 搜索 + 导航 + 命令合一 | 地址栏 / 输入框 | Chrome, Arc Browser |
| 搜索建议导航 | 输入时展示分类导航 | 搜索框聚焦 | Amazon, 淘宝, Google |

**设计要点**：

| 要素 | 最佳实践 |
|------|---------|
| 搜索位置 | 导航栏醒目位置，或支持全局快捷键触发 |
| 即时反馈 | 输入即搜索（debounce 200-300ms），不必按回车 |
| 结果分类 | 按类型分组显示（页面、功能、文档、人员等） |
| 最近/推荐 | 空状态展示最近访问和热门内容 |
| 键盘支持 | 上下键选择，回车确认，Esc 关闭 |
| 模糊匹配 | 支持拼音、缩写、近义词匹配 |

**常见错误**：
- ❌ 搜索是唯一导航方式 — 新用户不知道搜什么
- ❌ 搜索结果无分类，页面和功能混在一起
- ❌ 没有空状态引导（最近、推荐）
- ❌ 搜索响应慢，没有加载状态

**参考网站**：
- [Figma](https://figma.com) — Quick Actions (`Cmd + /`) 搜索命令和功能
- [VS Code](https://code.visualstudio.com) — Command Palette (`Ctrl+Shift+P`)
- [Linear](https://linear.app) — `Cmd + K` 全局搜索 + 命令
- [Notion](https://notion.so) — 全站搜索 + 最近页面
- [Algolia DocSearch](https://docsearch.algolia.com) — 文档网站的搜索式导航典范

### 4. 可折叠 / 手风琴菜单 (Collapsible / Accordion Menus)

**什么是可折叠菜单？**

通过展开/收起来渐进展示导航内容，用户只看到当前关注的分类，其余折叠隐藏。这是**渐进式披露 (Progressive Disclosure)** 原则的直接应用。

**结构示例**：
```
┌──────────────────────────┐
│ ▼ 产品管理               │  ← 已展开
│   ├ 产品列表             │
│   ├ 添加产品             │
│   ├ 分类管理             │
│   └ 库存管理             │
│ ▶ 订单管理               │  ← 已折叠
│ ▶ 用户管理               │  ← 已折叠
│ ▼ 数据分析               │  ← 已展开
│   ├ 销售报表             │
│   ├ 流量分析             │
│   └ 用户画像             │
│ ▶ 系统设置               │  ← 已折叠
└──────────────────────────┘
```

**适用场景**：
- 管理后台侧边栏
- 文档网站目录
- 移动端的多级导航
- 设置页面的分组

**展开模式**：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 单展开 (Exclusive) | 一次只展开一个分组 | 分组多、项数多，防止列表过长 |
| 多展开 (Independent) | 可同时展开多个分组 | 分组少，用户可能需要跨组操作 |
| 记忆展开 | 记住用户上次的展开状态 | 频繁使用的后台系统 |
| 默认展开 | 常用分组默认展开 | 提高常用功能可发现性 |

**设计要点**：
- 折叠/展开动画控制在 200-300ms，使用 ease-in-out 缓动
- 展开箭头 (▶/▼) 要清晰可见，点击区域覆盖整行
- 当前激活项所在的分组应自动展开
- 折叠状态下若有未读/通知，在分组标题上显示徽标

**常见错误**：
- ❌ 折叠后用户找不到之前访问的页面 — 应记忆展开状态
- ❌ 折叠动画过慢或有延迟 — 影响操作流畅感
- ❌ 所有分组默认折叠 — 新用户不知道里面有什么
- ❌ 展开后列表过长需要滚动 — 考虑使用单展开模式

**参考网站**：
- [Notion](https://notion.so) — 可无限嵌套的折叠侧边栏
- [VS Code](https://code.visualstudio.com) — 文件树的折叠/展开
- [Ant Design Pro](https://pro.ant.design) — 后台管理典型的折叠菜单
- [MDN Web Docs](https://developer.mozilla.org) — 文档目录的手风琴导航

### 5. 图标 + 提示方式 (Icon + Tooltip Approach)

**什么是图标+提示导航？**

使用纯图标展示导航项，鼠标悬停时显示文字提示（Tooltip）。适合需要极致空间效率的密集工具栏和可折叠侧边栏。

**结构示例**：
```
折叠状态：                    展开状态：
┌────┐                       ┌──────────────┐
│ 🏠 │ ← "首页"             │ 🏠 首页       │
│ 📊 │ ← "数据"             │ 📊 数据分析   │
│ 👥 │ ← "用户"             │ 👥 用户管理   │
│ 📦 │ ← "项目"             │ 📦 项目管理   │
│ ⚙️ │ ← "设置"             │ ⚙️ 设置       │
│    │                       │              │
│ ◀  │ 折叠/展开按钮         │ ◀ 折叠       │
└────┘                       └──────────────┘
  ~48px                          ~220px
```

**适用场景**：
- 可折叠侧边栏的收起状态
- 工具栏 / 工具面板
- 桌面应用的 Dock 栏
- 空间极其有限的界面

**设计要点**：

| 要素 | 最佳实践 |
|------|---------|
| 图标选择 | 使用通用、可识别的图标，避免抽象隐喻 |
| Tooltip 延迟 | 悬停 300-500ms 后显示，避免鼠标扫过时闪烁 |
| Tooltip 位置 | 侧边栏图标 → 右侧显示；工具栏 → 下方显示 |
| 激活状态 | 当前激活项用背景色或侧边指示条标识 |
| 可展开 | 始终提供展开为"图标+文字"的选项 |
| 无障碍 | 必须添加 `aria-label`，Tooltip 不是无障碍替代方案 |

**图标可识别性等级**：
```
高识别度（可单独使用）：  🏠 首页  🔍 搜索  ⚙️ 设置  👤 用户  🔔 通知
中等识别度（建议配文字）：📊 分析  📦 项目  📋 任务  🏷️ 标签  📁 文件
低识别度（必须配文字）：  🔮 AI功能  🧩 集成  🎯 OKR  💎 高级版
```

**常见错误**：
- ❌ 使用含义不明的自定义图标 — 用户完全猜不到功能
- ❌ 没有提供展开为文字模式的选项 — 新用户无法学习
- ❌ 忘记 `aria-label` — 屏幕阅读器用户完全无法使用
- ❌ 图标风格不统一 — 混用线性、填充、不同粗细的图标

**参考网站**：
- [Slack](https://slack.com) — 可折叠的图标侧边栏
- [Figma](https://figma.com) — 左侧工具栏的图标 + Tooltip
- [VS Code](https://code.visualstudio.com) — Activity Bar 的图标导航
- [Linear](https://linear.app) — 折叠侧边栏时的图标模式

### 6. 分类分组策略 (Categorization and Grouping)

**为什么分类如此重要？**

人类的短期记忆只能处理 **7±2 个信息块**（米勒定律）。当导航项超过这个数量，必须通过分组将其组织为可管理的信息块。

**分组方法**：

| 方法 | 说明 | 适用场景 | 示例 |
|------|------|---------|------|
| 功能分组 | 按功能类型聚合 | 后台管理 | 用户管理、内容管理、系统设置 |
| 用户角色分组 | 按目标用户聚合 | 多角色平台 | 买家中心、卖家中心、管理员 |
| 工作流分组 | 按使用流程聚合 | 项目管理工具 | 规划、执行、监控、复盘 |
| 字母排序 | 按首字母/拼音排列 | 术语表、通讯录 | A-Z 索引 |
| 频率分组 | 按使用频率排列 | 工具类应用 | 常用、全部、最近使用 |

**分组的视觉设计**：
```
┌──────────────────────────┐
│ 核心功能                  │ ← 分组标题（灰色小字）
│   📊 仪表盘              │
│   📋 任务                │
│   📦 项目                │
│                          │ ← 分组间距（8-16px）
│ 团队协作                  │ ← 分组标题
│   👥 成员                │
│   💬 消息                │
│   📅 日历                │
│                          │
│ ─────────────            │ ← 分隔线（可选）
│ 系统                     │
│   ⚙️ 设置               │
│   ❓ 帮助                │
└──────────────────────────┘
```

**设计要点**：
- 每组控制在 3-7 个项，组名简短明确（2-4 字）
- 分组之间用间距、分隔线或标题区分
- 最重要/最常用的分组放在顶部
- 系统类（设置、帮助）放在底部，形成稳定的视觉锚点
- 支持用户自定义分组和排序（高级功能）

**常见错误**：
- ❌ 分组标准不一致 — 有的按功能，有的按对象，逻辑混乱
- ❌ 单个分组内项数过多（>10 项） — 失去分组的意义
- ❌ 分组标题不明确 — "其他""更多""杂项" 都是糟糕的组名
- ❌ 分组之间没有视觉区分 — 看起来还是一长串列表

**参考网站**：
- [Notion](https://notion.so) — 工作区/私人/收藏的分组方式
- [Figma](https://figma.com) — 按图层类型和页面的组织方式
- [GitHub](https://github.com) — 仓库设置页面的分组导航
- [AWS Console](https://aws.amazon.com/console/) — 海量服务的分类分组

### 7. 优先级策略 (Prioritization Strategies)

**核心思想：不是所有导航项都需要同等展示。**

根据用户行为数据和业务优先级，对导航项进行分层展示。

**优先级分层模型**：
```
┌────────────────────────────────────────────────┐
│ 第 1 层：始终可见（3-5 项）                      │
│ 最高频使用的核心功能                              │
│ 示例：首页、搜索、通知、个人中心                   │
├────────────────────────────────────────────────┤
│ 第 2 层：一次交互可达（5-10 项）                  │
│ 下拉菜单、展开面板中的常用功能                     │
│ 示例：项目列表、团队、报告、集成                   │
├────────────────────────────────────────────────┤
│ 第 3 层：按需访问（10+ 项）                      │
│ 搜索、设置页、"更多"菜单中的低频功能               │
│ 示例：API 文档、导入导出、审计日志                 │
└────────────────────────────────────────────────┘
```

**常用优先级策略**：

| 策略 | 说明 | 实现方式 |
|------|------|---------|
| 数据驱动排序 | 根据点击率/使用频率排列 | 分析埋点数据，高频项置顶 |
| 渐进式披露 | 先展示核心，点击"更多"展开 | Priority+ 模式、展开按钮 |
| 上下文导航 | 根据当前页面/任务动态展示 | 不同页面显示不同的子导航 |
| 角色定制 | 不同用户角色看到不同导航 | 权限系统控制可见性 |
| 个性化导航 | 用户可自定义常用入口 | 收藏/固定功能、拖拽排序 |
| 最近使用 | 显示用户最近访问的入口 | 记录访问历史，快捷回访 |

**"更多"菜单的正确做法**：
```
✅ 正确                              ❌ 错误
┌──────────────────┐                ┌──────────────┐
│ 首页 任务 项目    │                │ 首页  更多 ▾  │
│ 团队  [更多 ▾]   │                │              │
│  ┌─────────┐     │                │ （所有功能都 │
│  │ 报告    │     │                │  藏在更多里） │
│  │ 日历    │     │                └──────────────┘
│  │ 集成    │     │
│  │ 设置    │     │
│  └─────────┘     │
└──────────────────┘
主要项外露，次要项收纳               主要项也被隐藏
```

**常见错误**：
- ❌ 所有导航项平等展示 — 用户面临选择困难
- ❌ 只凭主观判断优先级 — 应结合数据分析
- ❌ 将"更多"作为一级导航入口，里面塞满内容
- ❌ 不提供自定义能力 — 不同用户有不同的高频操作

**参考网站**：
- [Slack](https://slack.com) — 频道可置顶、收藏，按优先级分区
- [Gmail](https://mail.google.com) — 主要/社交/推广的智能分类标签
- [Jira](https://www.atlassian.com/software/jira) — 可自定义的侧边栏快捷入口
- [GitHub](https://github.com) — 仓库页的 Priority+ 标签栏

### 8. 大型菜单的响应式设计

大型导航在不同屏幕尺寸下需要完全不同的展现形式。

**响应式转换策略**：

```
桌面端 (≥1024px)           平板端 (768-1023px)        移动端 (<768px)
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│                  │     │                  │     │                  │
│ 超级菜单         │     │ 精简下拉菜单     │     │ 全屏分类列表     │
│ 多列平铺展示     │ ──→ │ 单列滚动         │ ──→ │ 手风琴式折叠     │
│                  │     │ 隐藏图片/描述    │     │ 搜索栏置顶       │
│                  │     │                  │     │                  │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

**各模式的转换规则**：

| 桌面端模式 | 平板端转换 | 移动端转换 |
|-----------|-----------|-----------|
| 超级菜单（多列） | 减少列数（2-3 列），隐藏图片 | 全屏列表 + 手风琴折叠 |
| 级联飞出菜单 | 改为单列展开/收起 | 逐级钻入 (drill-down) 导航 |
| 图标+文字侧边栏 | 自动折叠为纯图标 | 转为底部导航 + 汉堡菜单 |
| 搜索式导航 | 保持不变，优化键盘 | 全屏搜索覆盖层 |
| 分组侧边栏 | 保持，可折叠 | 转为抽屉式菜单 |

**移动端"逐级钻入"模式**：
```
第 1 级                    第 2 级                    第 3 级
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ [✕]   全部分类   │     │ [←]   电子产品   │     │ [←]    手机      │
├──────────────────┤     ├──────────────────┤     ├──────────────────┤
│                  │     │                  │     │                  │
│ 电子产品      ▸  │ ──→ │ 手机          ▸  │ ──→ │ iPhone          │
│ 服装鞋帽      ▸  │     │ 电脑平板      ▸  │     │ Samsung         │
│ 家居生活      ▸  │     │ 智能穿戴      ▸  │     │ 华为            │
│ 食品生鲜      ▸  │     │ 摄影器材      ▸  │     │ 小米            │
│ 图书影音      ▸  │     │ 游戏设备      ▸  │     │ OPPO            │
│                  │     │                  │     │                  │
└──────────────────┘     └──────────────────┘     └──────────────────┘
   点击分类进入 →           ← 返回按钮回退           ← 返回按钮回退
```

**设计要点**：
- 移动端避免悬停触发 — 触摸屏没有悬停状态
- 逐级钻入时提供明确的返回按钮和层级标题
- 移动端为搜索框提供显著入口，减少浏览层级的需要
- 转场动画使用左右滑动，暗示层级深入/返回方向

**常见错误**：
- ❌ 移动端直接显示桌面端的超级菜单 — 完全无法使用
- ❌ 逐级钻入没有返回按钮 — 用户被困在深层级
- ❌ 移动端菜单没有搜索 — 浏览大量分类极其痛苦
- ❌ 响应式断点处导航突然消失或布局跳变

### 9. 真实案例分析

#### 案例 1：Amazon — 电商超级菜单的标杆

```
策略组合：侧滑超级菜单 + 级联菜单 + 搜索 + 面包屑
```

- **搜索栏**：导航栏最醒目位置，支持分类筛选 + 自动补全
- **超级菜单**：左侧滑出，展示所有商品部门
- **级联子菜单**：悬停商品部门展开详细分类（安全三角）
- **面包屑**：商品页显示完整路径，方便回溯
- **个性化**："根据浏览历史"推荐的快捷入口

**设计亮点**：搜索和分类浏览两条路径并重，用户自由选择。

#### 案例 2：阿里巴巴/淘宝 — 中国电商导航模式

```
策略组合：分类超级菜单 + 标签导航 + 搜索 + 个性化推荐
```

- **顶部搜索**：极大的搜索框占据导航核心位置
- **分类菜单**：悬停弹出多级分类面板
- **标签导航**：搜索下方的热门分类标签快捷入口
- **个性化**："猜你喜欢"的智能导航区域

**设计亮点**：搜索权重极高，配合个性化推荐减少浏览负担。

#### 案例 3：Figma — 专业工具的搜索式导航

```
策略组合：图标工具栏 + 搜索命令面板 + 分层菜单
```

- **左侧工具栏**：图标 + Tooltip 的密集工具栏
- **Quick Actions**：`Cmd + /` 搜索任何功能和命令
- **右键菜单**：上下文相关的级联菜单
- **顶部菜单栏**：传统的分类下拉菜单（文件/编辑/视图...）

**设计亮点**：新手可通过菜单逐项浏览，熟练用户用搜索直达。

#### 案例 4：AWS Console — 海量服务的导航挑战

```
策略组合：搜索 + 分类分组 + 收藏固定 + 最近使用
```

- **200+ 服务**：传统导航完全无法覆盖
- **搜索**：顶部全局搜索是最主要的导航方式
- **分类分组**：服务按"计算""存储""数据库"等分组
- **收藏栏**：用户可将常用服务固定到导航栏
- **最近访问**：自动记录最近使用的服务

**设计亮点**：承认浏览不可行时，搜索 + 收藏 + 最近使用三管齐下。

### 大型导航策略选择指南

| 导航项数量 | 推荐策略 | 组合搭配 |
|-----------|---------|---------|
| 10-20 项 | 超级菜单 或 分组侧边栏 | + 面包屑 |

| 50-100 项 | 级联菜单 + 搜索 | + 收藏/最近 + 面包屑 |

**决策树**：
```
你的导航项有多少？
├── 10-20 项
│   ├── 内容型网站 → 超级菜单（分类 + 图片预览）
```

## 本章要点

- **选择合适的导航类型**：根据产品类型、内容量和目标平台选择
- **控制导航层级**：不超过 3 层，遵循三次点击原则
- **标签命名要具体**：使用用户语言，避免模糊词汇
- **响应式是必须的**：为不同屏幕尺寸设计适配方案
- **无障碍不可忽视**：语义化 HTML、键盘导航、ARIA 属性
- **明确当前位置**：用户应随时知道自己在哪里
- **移动端考虑拇指热区**：核心操作放在屏幕下半部分
